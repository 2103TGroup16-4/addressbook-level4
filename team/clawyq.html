PROJECT: CLInic

Introduction

This Project Portfolio Page documents `clawyq’s contributions to the development of CLInic.

CLInic is a desktop application made to manage and expedite operations in clinics. It was inspired by the Command Line Interface (CLI), which allowed users to issue commands to the computer efficently. However, the CLI’s non-visual display and walls of cryptic text makes it intimidating to the general, non-tech-savvy populous.

These reasons were the main impetus behind CLInic, which marries the simplistic design and efficient command issuing mechanism of the CLI with an intuitive Graphical User Interface (GUI). This culminated in a clean application that is easy for anyone to use, all while providing the powerful functionalities and efficiencies provided by a CLI.

Below are some of the key functionalities provided by CLInic:

  • Allows users to manage patient and medicine information

  • Allows users to easily generate documents to be issued by a clinic like Medical Certificates, Receipts and Referral Letters efficiently

  • Provides intuitive workarounds to normally arduous but common administrative tasks such as restocking and dispensing medicine

CLInic is the product of semester-long efforts of a team of five students from NUS School Of Computing as part of the CS2103T Software Engineering module. Through the course of the module, the team morphed a very primitive application, which had basic add/delete functionalities akin to that of a phone/contact book, into a fully-fledged clinic management application. The main features of the application is primarily coded in Java while the GUI is handled by JavaFX, HTML and CSS.

I played a major role in the development of the Document class, the relevant Receipt, MedicalCertificate and Referral classes that extend it and the relevant patient management operations that are required for them to work.

Document Generation

Generating medical certificate: mc

Generates medical certificate for a Served Patient.
Format: mc INDEX

  • Generates a medical certificate for the patient at the specified INDEX. The index refers to the index number shown in the Served Patient queue. The index must be a positive integer (i.e. 1, 2, 3, …​).

Examples:

  • mc 1
    Generates a medical certificate for the 1st patient in the Served Patient queue.

Document

Current Implementation

The Document class represents all documents that can be issued by the clinic. The Receipt, MedicalCertificate and ReferralLetter classes all inherit from it.
It specifies what general information these documents should have and how they should be formatted. It can be thought of as the formatting template for all the information belonging to its inheriting classes.

Implementation of Receipt

The Receipt class structure is as follows:

  • has attributes of the Served Patient it is generated from, which includes the served patient’s name and IcNumber.

    • These attributes are marked as final as they should not be changed.

  • contains all the Medicine and their associated MedicineQuantity.

    • These are the medicines and their quantities respectively, prescribed by the doctor to the patient during his consultation.

Users can generate a Receipt for a specific ServedPatient by invoking the ReceiptCommand accompanied by the served patient’s index in the ServedPatientList.

Design Considerations

Aspect: Implementation of the Document

  • Alternative 1 (current choice): Using a typical class to implement Document.

    • Pros:

      • Allows all classes inheriting from Document to use a standardised method to format the relevant information that appears in every document that is issued by the clinic.

    • Cons:

      • Java does not support multiple inheritance; therefore, it prevents the classes that are inheriting from Document from inheriting from other useful classes or becoming an enum.

      • Comparatively, inheriting from classes is more computationally expensive than using interfaces due to the overhead incurred looking up methods and various class members.

  • Alternative 2: Using an interface to implement Document.

    • Pros:

      • The classes implementing Document can be converted to enums, which ensures that the inputs fed to these classes are valid.

        • For example, an enum will prevent the issuing of a Medical Certificate that excuses a person from work/school for 1,000,000 days!

    • Cons:

      • Unable to standardise the way the implementing classes implement the generate method.

        • This can potentially allow the inheriting classes to omit certain fields that are relevant to the printing of a document, such as date and time of the consultation.

        • Even if all the information are present, they might be formatted differently.
